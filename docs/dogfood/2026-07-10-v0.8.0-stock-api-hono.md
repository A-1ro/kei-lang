## お題

在庫参照 API 2 エンドポイント: GET /stock/{itemId} と POST /stock/{itemId}/adjust。
- 4 分岐の HTTP 応答(200/400/404/422)
- 「調整結果は必ず非負」を ensures で明示
- HTTP モデルと shape check の使用

## 結果: 8/10 — 4 ファイル初見エラーゼロ、修正ループゼロ

- **契約表現の深化**: `computeAdjustResult` の非負契約を `result.valid implies result.count >= 0` と `result.count >= 0 implies result.valid` の**双方向 implies**で表現(初見で自然にこの形に到達)
- **参照**: SKILL.md ×1、examples(stock_direct / borrow / inventory / records / options / enums / matching)、spec は grep 3 回のみ
- kei check ×5、kei fmt --check ×4、いずれも exit 0

## 観測されたギャップ(2 件、いずれも v0.9 設計への実需入力)

### ギャップ 1: パスパラメータ抽出手段が未文書化

- 現状: HttpRequest は `path: String`(生のフルパス)のみ。String stdlib に `split`/`indexOf`/`substring` がなく、Kei 側で `/stock/42` から `"42"` を切り出せない
- エージェントの回避策: 「TS 側 Hono の path param 抽出済みの値をハンドラに追加引数で渡す」と推測仮定
- 実際の @kei/hono の mount() はこの追加引数シグネチャに未対応
- **v0.9 で決めるべき**: (a) mount() の handler シグネチャを拡張(`(req, params: Map<String, String>) => ...` など)、(b) Kei 側 String に `split(delimiter)` を追加(v0.5 M30 の伏線で予告されていた本格 String API)、(c) HttpRequest に `pathParams: Map<String, String>` を追加、のどれか

### ギャップ 2: 依存注入(Map<String, Int> 在庫マスタ)の経路が未文書化

- 同じく追加引数で仮定(context / env / bindings のパターンが SKILL.md にない)
- v0.9 で Workers bindings(KV/D1)対応を設計する際に必ず解く必要あり

## 総評

8/10。**v0.8 の総合実力**(v0.7 の async・v0.6 の extern・v0.5 の Map/spread/lambda を全部組み合わせる複合お題)を初見でエラーゼロ通過し、双方向 implies 契約という洗練された表現まで到達。減点はアダプタ層の実配線(パスパラメータ・依存注入)が未文書化で仮定を置くしかなかった点 — これは **v0.9 の設計インプット**として非常に価値がある実需。v1.0 の実運用性を左右する。
