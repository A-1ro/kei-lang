# KEI-E2011: invalid Map key type

型検査: `Map<K, V>` の `K`(キー型)に許されない型を書いた。v0.5 の `Map` は段階1
(spec/kei-spec-v0.3-collections.md §7)で、キーは `Int` / `String`、またはそれらを基底に持つ
tagged 型のみを許す。それ以外(`Record` / `enum` / `List<T>` / `Option<T>` / `Result<T, E>` /
`Bool` など)をキーにするのは静的エラー。

```kei
let bad: Map<Bool, Int> = Map.empty()
// error: Map key type must be 'Int', 'String', or a tagged type over them; found 'Bool'
```

## なぜ制限するか

`Map` のキー同値性はトランスパイル先の TS `Map`(SameValueZero 比較)にそのまま委譲する。
`Int` / `String`(+ tagged 基底)はスカラーの値同値性と一致するため安全に委譲できるが、
`Record` などの合成型をキーにすると「参照同一性」と「構造同値性」の混同(`==` の制約 §5.1・
KEI-E2010 と同種の問題)がキー照合にも及ぶ。段階1ではこれを持ち込まない。

## 修正

- tagged 型でキーを表す(例: `type ProductId = String tagged "ProductId"`)。
- 合成型をキーにしたい場合は、スカラーのフィールドを取り出してキーにする
  (例: `product.id`(String)をキーにし、値側にレコード全体を持たせる)。

キー型を任意の等値比較可能型へ拡張するかは、段階2の量化・等価性設計と合わせて再評価する
(spec/kei-spec-v0.3-collections.md §7.7)。
